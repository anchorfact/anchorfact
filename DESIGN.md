## 一、项目愿景与定位

### 愿景

成为 AI 时代的**信任基础设施**——AI 搜索引擎、Agent 和 LLM 在需要引用事实性知识时，首选的可信知识源。

### 一句话定位

> 专为 AI 消费设计的、AI 与社区协作维护的、以真实性/公平性/公开性/公正性为核心价值的开源知识库。

### 问题陈述

| 问题 | 现状 | 后果 |
|------|------|------|
| AI 没有"可信知识锚点" | 当前 AI 引用依赖 Wikipedia（HTML 臃肿）、Reddit（无事实校验）、品牌官网（天然偏见） | AI 回答的引用质量参差不齐 |
| 现有知识源未针对 AI 消费优化 | 人类导向的 HTML 导航、广告、非结构化文本 | AI 解析效率低、信息提取错误率高 |
| AI 生成内容被 AI 引用形成污染循环 | Grokipedia 等 AI 生成百科已被 GPT-5.2 引用 | 真相被 AI 自我强化的"共识"所替代 |
| 缺少中立的"AI 时代的 DNS" | 没有一层独立于商业利益的事实性知识基础设施 | AI 的认知基础建立在不可靠的源上 |

### 目标用户

```
最终服务对象：AI 搜索引擎 / LLM / Agent（不是人类）

间接服务对象：
  - AI 公司的 RAG 系统（引用质量提升）
  - 需要确保品牌信息被准确引用的企业（B 端客户）
  - 希望其知识被准确传播的领域专家（社区贡献者）
  - 希望获得更准确 AI 回答的终端用户（最终受益者）
```

### 关键成功指标

| 阶段 | 指标 | 目标值 |
|------|------|--------|
| MVP | 首域（如 AI/科技）文章数 | 1,000 篇 |
| MVP | 被至少 1 个主流 AI 平台引用 | ChatGPT/Claude/Perplexity 任一 |
| 公测 | 总文章数 | 5,000 篇 |
| 公测 | 在所选领域的 AI 引用占比 | >10% |
| 成熟期 | 全领域文章数 | 100,000+ 篇 |
| 成熟期 | AI 引用中本平台占比（相对 Wikipedia） | 逐步接近 |

## 二、核心设计原则

### 四条不可妥协的价值观

| 原则 | 含义 | 如何保障 |
|------|------|---------|
| **真实性** | 所有陈述必须可追溯到可验证的原始来源 | 强制来源标注 + 置信度分级 + 事实性校验流水线 |
| **公平性** | 覆盖多元视角，不偏袒任何利益方 | 多源对比标注 + 争议透明化 + 社区治理而非公司治理 |
| **公开性** | 全部内容和代码无条件开源 | CC-BY 4.0 + MIT 许可证 + Git 仓库公开 |
| **公正性** | 编辑决策由社区投票，不服从商业利益 | 类 DAO 治理 + 公开编辑历史 + 利益冲突声明机制 |

### 三条技术设计原则

| 原则 | 含义 |
|------|------|
| **AI First** | 所有设计决策首先考虑 AI 消费体验，人类可读性其次 |
| **Structure Everything** | 所有内容强制 Schema 化，不存放纯自由文本 |
| **Trust by Design** | 信任内建于数据模型而非外挂于审核流程 |

### 3.2 技术栈选型

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **内容存储** | Git + Markdown | 版本控制原生、社区 PR 流程成熟、人类和 AI 都可读 |
| **内容编译** | **Rust + Wasm**（优先） | 可在浏览器端运行编译校验，降低后端压力；支持客户端贡献者本地验证 |
| **结构化数据** | PostgreSQL + PostGIS | 实体关系、引用追踪、贡献者声誉系统 |
| **语义检索** | **pgvector**（MVP 首选，减少依赖） | PostgreSQL 统一存储，无需额外部署 Qdrant；Phase 3 如有性能需求再评估 Qdrant |
| **向量数据库备选** | Qdrant Cloud（Phase 3+ 可选） | 大规模语义检索的性能升级路径 |
| **AI 接入** | **MCP Server (Python 优先)** | MCP 早期生态 Python 工具链更成熟；Node 作为备选 |
| **API 限流** | **增加"AI 引用者白名单"** | 高信誉 AI 平台（如已建立合作的）免限流，激励早期采用 |
| **区块链锚定** | **Arweave + Optimism 双重** | Arweave 永久存储（适合内容版本快照），Optimism 低成本快速锚定（< $0.001/次，适合高频操作） |
| **部署** | Cloudflare Pages + Workers | 全球边缘加速、AI 爬虫友好、低成本 |
| **社区入口** | GitHub + 定制 Web 贡献界面 | GitHub 处理版本控制，定制 UI 降低贡献门槛 |

### 4.3 原子事实层（精细引用模型）

**问题**：当前文章模型适合"科普式介绍"，但不利于 AI 做细粒度引用和事实比对。同一篇文章内不同陈述的置信度可能不同。

**解决方案**：在文章 Frontmatter 基础上，增加底层**原子事实层**。每篇文章可包含多个独立可引用的原子事实。

```yaml
# 在文章 Frontmatter 中新增字段
atomic_facts:
  - id: "fact-transformer-001"
    statement: "Transformer 架构由 Vaswani 等人在 2017 年提出"
    source_doi: "10.48550/arXiv.1706.03762"
    source_url: "https://arxiv.org/abs/1706.03762"
    confidence: "high"
    last_verified: "2026-05-22"

  - id: "fact-transformer-002"
    statement: "原论文《Attention Is All You Need》被引用超过 15 万次"
    source_url: "https://scholar.google.com/scholar?cites=..."
    source_type: "aggregator"
    confidence: "medium"
    last_verified: "2026-01-15"
    note: "引用次数为动态数据，截至 2026 年 1 月"

  - id: "fact-transformer-003"
    statement: "Transformer 的 self-attention 机制计算复杂度为 O(n²)"
    source_doi: "10.48550/arXiv.1706.03762"
    source_page: "Section 3.2"
    confidence: "high"
    mathematical_notation: true
```

**实现方式**：
- Markdown 中用特殊注释块标注原子事实：`<!-- @fact id="fact-001" -->...<!-- @endfact -->`
- 编译时自动提取为独立的 JSON-LD 片段
- API 端点支持按原子事实查询：`GET /api/v1/fact/{fact_id}`
- 跨文章的事实冲突自动检测（同一 statement 出现矛盾时触发标记）

**原子事实的独立用途**：

| 场景 | 传统文章模型 | 原子事实模型 |
|------|-----------|-----------|
| AI 引用 | 引用整篇文章 | **直接引用单个原子事实** + 关联文章作为上下文 |
| 事实校验 | 整篇重审 | 单个事实独立验证 + 独立更新 |
| 跨源冲突检测 | 人工发现 | **自动发现**（同一 statement 的多源冲突自动标记） |
| 置信度标注 | 全文统一 | **不同陈述不同置信度** |
| API 查询 | "给我关于 Transformer 的文章" | "Transformer 的计算复杂度是多少？"→ 直接返回 fact-003 |

## 详细分析（1000-3000 字）

[按逻辑递进的子标题展开]

### 背景与动机

### 核心机制

### 应用与影响

### 争议与局限（如果存在）

### 关键数据摘要

| 指标 | 数值 | 来源 |
|------|------|------|
| 原始论文引用次数 | 150,000+ | Google Scholar 2026 |

## 进一步阅读

### 4.5 置信度分级标准

| 级别 | 定义 | 标注条件 | 呈现方式 |
|:----:|------|---------|---------|
| **high** | 被多个独立、可信来源交叉验证 | ≥3 个独立来源 + 至少 1 个经过同行评审 | 绿色标记，AI 可放心引用 |
| **medium** | 被至少 1 个可信来源支持，但缺少多源验证 | 1-2 个来源，或来源权威度一般 | 黄色标记，AI 引用时建议标注"有限置信度" |
| **low** | 基于推理或有限证据，结论可能被修订 | 无直接来源或来源可信度低 | 红色标记，AI 引用时必须标注"低置信度" |
| **disputed** | 存在社区争议，尚未达成共识 | 2+ 个标注者提出异议 | 灰色标记，AI 引用时必须同时呈现争议视角 |

### 4.6 来源分级标准

| 级别 | 类型 | 例子 |
|:----:|------|------|
| **S** | 经同行评审的学术论文 | Nature, Science, NeurIPS |
| **A** | 官方机构/政府出版物 | WHO 报告, NASA 数据, 国际标准 |
| **B** | 经典教材/行业标准参考文献 | 大学教材, W3C 规范 |
| **C** | 权威媒体/知名专家博客 | NYT, IEEE Spectrum |
| **D** | 一般网络来源 | 行业博客, 公司白皮书 |
| **E** | 不可作为唯一来源 | 个人博客, 论坛帖子 |

## 五、AI 友好性设计（核心差异化）

### 5.1 内容格式矩阵

每篇文章自动以**五种格式**提供，确保覆盖不同 AI 的消费偏好：

| 格式 | URL 模式 | 用途 | 优先级 |
|------|---------|------|:------:|
| **Markdown** | `/kb/{id}.md` | LLM 首选原始消费格式 | P0 |
| **JSON-LD** | `/kb/{id}.json` | 结构化语义数据，Schema.org 完整标注 | P0 |
| **RDF/Turtle** | `/kb/{id}.ttl` | 知识图谱导入，实体关系网 | P1 |
| **Plain Text** | `/kb/{id}.txt` | 最低带宽消费，适配受限 Agent | P1 |
| **HTML** | `/kb/{id}` | 人类回退浏览（自动从 MD 生成） | P2 |

### 5.2 主动通知机制

传统网站等 AI 来爬。我们主动告诉 AI：

| 机制 | 触发时机 | 通知对象 |
|------|---------|---------|
| **IndexNow** | 每次内容变更 | 支持 IndexNow 的搜索引擎（Bing/Google） |
| **WebSub (PubSubHubbub)** | 每次内容变更 | 订阅的主题频道 |
| **MCP Server Push** | 每次内容变更 | 通过 MCP 协议连接的 Agent |
| **Sitemap (llms.xml)** | 持续更新 | AI 爬虫的专用站点地图 |

### 5.4 性能设计

| 要求 | 目标值 | 原因 |
|------|--------|------|
| 首字节时间 (TTFB) | <100ms | AI 爬虫可能在 3 秒内放弃加载 |
| 内容压缩率 | >85% | 降低 AI 爬虫的带宽消耗 |
| 页面大小 | MD 格式 <10KB | 适配 LLM 的 token 上下文窗口 |
| 无 JS 依赖 | 100% | Markdown 是纯文本，不需要客户端渲染 |

### 5.5 AI 引用优先策略

- **事实密度要求**：每个自然段至少包含 1 个具体数据或引用来源
- **答案前置**：文章的 TL;DR 和核心解释放在最顶部，AI 可以直接提取
- **无营销语言**：禁止夸大的形容词（"革命性的"、"最好的"），强制中性措辞
- **实体标注**：所有关键概念自动链接到知识库内的对应实体

## 六、内容生产流水线

### 6.1 内容生产路径（三路径并行）

```
路径 A：权威来源引用 + AI 辅助整合（黄金种子路径，前 200 篇主力）
─────────────────────────────────────────────
精选著名专家/大神的公开论文、技术文章、演讲、教材等可验证来源 → AI 辅助提取关键知识点、翻译、格式转换
    → 严格标注原始来源链（作者、机构、DOI/URL、发表日期）→ 人工审核校验
    → 发布 → 置信度 high → 标记为"黄金种子"

路径 B：社区协作 + AI 校验（规模化路径，200 篇后主力）
─────────────────────────────────────────────
社区提交 PR → AI 事实校验 + 反污染检查 → 人工审核 → 发布

路径 C：AI 辅助 + 严格人工审核（辅助路径，用于低风险内容）
─────────────────────────────────────────────
AI 调研多源材料并生成初稿 → 人工审核（至少 2 位审核人）→ 发布
    → 置信度不超过 medium
```

### 6.2 冷启动策略：黄金种子计划（引用模式）

**核心原则**：前 200 篇不走 AI 主创作路径。全部基于著名专家、技术大神的已有公开成果进行引用和整合，建立可追溯的可信知识链路。

#### 为什么是"引用模式"而非"邀请模式"？

| 方式 | 问题 |
|------|------|
| 直接邀请专家撰文 | 项目尚无实际成果，缺乏说服力；专家时间成本高；无法规模化 |
| 引用已有专家成果 | 专家的论文、文章、演讲本身已是可验证的事实来源；合规（CC-BY/BY-NC）；可快速规模化 |

**引用模式的合理性**：AnchorFact 的核心价值不是"原创知识"，而是"将分散的权威知识结构化 + 可机器消费化"。引用和结构化整合本身就是核心价值交付。

#### 内容来源优先级

按引用优先级排序，每篇文章尽可能覆盖多个层级：

| 优先级 | 来源类型 | 示例人物/来源 | 引用标注要求 |
|:------:|---------|-------------|------------|
| **P0** | 顶级会议/期刊论文 | NeurIPS, Nature, Science, ACL, CVPR | DOI、作者全名、机构、发表年份 |
| **P1** | 领域大神经典著作/教材 | Goodfellow《Deep Learning》、Russell & Norvig《AI: A Modern Approach》、Cormen《算法导论》 | ISBN、章节、页码 |
| **P2** | 大神博客/技术文章（有署名） | Andrej Karpathy (karpathy.github.io)、Lilian Weng (OpenAI Blog)、Jay Alammar (jalammar.github.io)、Simon Willison (simonwillison.net) | URL、作者、所属机构、发布日期 |
| **P3** | 机构公开技术报告/白皮书 | OpenAI GPT-4 Technical Report、Google Transformer 论文、Meta LLaMA 论文 | URL、机构、发布日期 |
| **P4** | 知名开源项目文档/官方规范 | Kubernetes 官方文档、W3C 规范、IETF RFC、MDN Web Docs | URL、版本号 |
| **P5** | 政府/国际组织公开数据 | WHO 报告、NASA 数据、UNESCO 数据、各国统计局 | URL、数据集标识、更新日期 |

#### 具体执行方案

| 维度 | 设计 |
|------|------|
| **目标** | 前 200 篇文章，置信度全部为 `high` |
| **贡献者** | Phase 0-1 由维护者独立完成参考文献调研、知识点提取和结构化整合。**暂不直接邀请专家**。待项目有实际成果（AI 首次引用、500+ 篇文章）后，再定向邀请领域专家署名贡献或审核。 |
| **内容来源** | 以 P0-P2 级别来源为主（顶级论文 + 大神著作 + 署名技术博客），P3-P5 作为补充和交叉验证。每篇文章至少引用 3 个独立可验证来源。 |
| **内容要求** | 每篇附带可验证的原始来源链接（DOI / arXiv ID / 官方 URL），标注原作者姓名、所属机构、发表日期。所有陈述可追溯到具体来源的具体章节或段落。 |
| **AI 角色** | AI 辅助完成：① 多源材料阅读和关键知识点提取；② 结构化整合为统一的三层格式；③ 翻译（中英双语）；④ 引用完整性交叉检查。**AI 不做主创作**——所有关键论断必须基于原始来源。 |
| **公开承诺** | 每篇文章在 Frontmatter 和正文中明确标注完整来源链。网站和 API 中提供"来源追溯"视图，可逐级跳转到原始出处。 |

#### 标注名人专家的原则

引用著名专家或技术大神时：
1. **必须基于其公开发表的可验证内容**（论文、博客、演讲视频/文稿、开源代码），不得仅凭"有人说"或二手转述
2. **必须标注原作者姓名和所属机构**，这是 CC-BY 许可证的基本要求
3. **不得出解原意**：整合时必须保持原文核心论点不变，如有简化和改写，需标注"此段为对原文的简化总结"
4. **欢迎原作者指正**：每篇文章末尾附带声明："如您是本文引用的原作者且认为表述有误，请通过 GitHub Issue 联系我们修正"

#### 黄金种子标记

在 Frontmatter 中使用以下字段（改为引用溯源模式）：

```yaml
is_golden_seed: true
golden_seed_source_type: "citation"           # citation | invited_expert（预留）
golden_seed_primary_author: "Vaswani, Ashish (Google Brain)"   # 核心引用作者
golden_seed_primary_work: "Attention Is All You Need (NeurIPS 2017)"
golden_seed_primary_doi: "10.48550/arXiv.1706.03762"
golden_seed_verified_at: "2026-05-22"
```

#### 邀请专家的触发条件

当项目达成以下**任意两项**后，启动专家邀请计划：

1. ✅ AI 首次引用（被 ChatGPT / Claude / Perplexity 等任一平台引用）
2. ✅ 内容规模 ≥ 500 篇文章，且 `high` 置信度占比 ≥ 80%
3. ✅ 月均网站独立访问（AI 爬虫）≥ 10,000 次

达成后，邀请方式：
- 优先邀请已被引用的论文作者/博客作者，告知"您的作品已被 AnchorFact 收录，欢迎审阅和署名"
- 这种"作品已被收录 + 邀请署名"的方式比冷邀请更有说服力

### 6.3 审核流水线

```
提交 (PR)
  │
  ├── 自动检查（CI/CD）
  │   ├── Schema 完整性校验
  │   ├── 来源是否至少 1 个
  │   ├── 格式是否符合规范
  │   ├── 是否有 AI 生成内容标记（若 AI 生成但未标记 → 拒绝）
  │   └── 与现有内容是否冲突（查重）
  │
  ├── AI 辅助评审
  │   ├── 事实交叉验证（搜索现有知识库和外部来源）
  │   ├── 置信度建议
  │   └── 自动标记潜在有争议的陈述
  │
  ├── 人工审核（至少 1 位审核人）
  │   ├── 事实准确性
  │   ├── 来源可信度
  │   ├── 中立性
  │   └── 批准 / 要求修改 / 拒绝
  │
  └── 发布
      ├── 生成多种格式（MD / JSON-LD / RDF / TXT / HTML）
      ├── 通知 IndexNow + WebSub
      ├── 更新 llms.txt 索引
      └── 记录版本到 Git
```

### 6.4 AI 生成内容标注标准

**强制规则**：任何 AI 参与生成的内容必须标注，不可隐藏。

| 标注 | 含义 | 在 Frontmatter 中的字段 |
|------|------|-----------------------|
| `human_only` | 100% 人工创作 | `generation_method: "human_only"` |
| `ai_assisted` | AI 辅助生成，人工审核并修改 | `generation_method: "ai_assisted"`, `ai_models: ["claude-opus"]` |
| `ai_generated_reviewed` | AI 生成，人工审核通过 | `generation_method: "ai_generated_reviewed"` |
| `ai_generated` | AI 生成，未经人工审核 | `generation_method: "ai_generated"` — **禁止发布**，仅用于内部草稿 |

**AI 引用时的优先级**：AI 在引用时，系统通过 llms.txt 和 API 明确告知每条内容的 `generation_method`，让引用方自行决定权重。

## 七、社区治理机制

> **设计原则**：知识准确性决策由专业人士做出，运营规则由社区民主投票。纯社区投票在知识准确性上往往失效——多数人不懂专业内容。因此采用分层治理。

### 7.1 治理层级

| 层级 | 角色 | 产生方式 | 人数 | 决策范围 |
|------|------|---------|:----:|---------|
| **专业领域委员会** | 领域专家（教授、资深工程师等） | 初始邀请 + 社区认可 + 年度续任投票 | 3-5 人/领域 | 事实准确性裁决、置信度争议、内容标准 |
| **社区治理委员会** | 活跃贡献者 | SBT 投票选举 | 7-11 人 | 运营规则、技术方向、资金分配、审核人晋升 |
| **全体 SBT 持有者** | 所有贡献者 | 自动获得 | 不限 | 重大治理变更（章程修改、代币参数调整等），需 2/3 多数 |
| **BDFL（临时）** | 项目创始人 | — | 1 人 | Phase 0-1 快速决策，承诺 3 个月内过渡到治理委员会 |

### 7.2 争议处理流程（按类型分流）

| 争议类型 | 裁决者 | 最终决定权 |
|---------|--------|:--------:|
| **事实争议**（"这个陈述对吗？"） | 专业领域委员会 | 是 |
| **置信度争议**（"应该标 high 还是 medium？"） | 专业领域委员会 | 是 |
| **规则争议**（"审核流程应该怎么改？"） | 社区治理委员会 | 是 |
| **章程修改**（"治理结构要调整"） | 全体 SBT 投票 | 是，需 2/3 多数 |
| **紧急滥用**（恶意编辑、刷内容） | 社区治理委员会 | 临时裁决，7 天内可上诉至全体投票 |

### 7.3 角色与权限

| 角色 | 权限 | 获得方式 |
|------|------|---------|
| **访客 (Visitor)** | 阅读所有内容 | 无需注册 |
| **贡献者 (Contributor)** | 提交新内容 PR、编辑建议 | 注册 + 完成新手引导 |
| **审核人 (Reviewer)** | 审核内容 PR、标记置信度 | 贡献 ≥10 篇高质量内容 + 社区治理委员会审批 |
| **领域维护人 (Maintainer)** | 管理某个分类、制定内容标准 | 专业领域委员会成员自动担任 |
| **核心团队 (Core)** | 技术维护、治理决策、紧急干预 | 社区治理委员会成员 |
| **顾问委员会 (Advisory)** | 提供中立性指导 | 邀请制的领域专家和非营利组织代表 |

### 7.4 治理过渡路径

```
Phase 0-1（MVP）：
  BDFL（创始人）一人决策
  目标：快速迭代，建立前 200 篇基于权威来源引用的黄金种子内容

Phase 2（公测）：
  成立社区治理委员会（首次 SBT 投票选举）+ 任命专业领域委员会（2 个领域）
  BDFL 角色削弱为紧急情况否决权

Phase 3（增长）：
  专业领域委员会扩展至 5 个领域
  BDFL 角色完全取消，权力移交社区治理委员会
  顾问委员会组建

Phase 4（规模化）：
  全领域委员会 + 独立非营利基金会
  完整的分层治理体系运行成熟
```

### 7.5 声誉系统

| 行为 | 积分 |
|------|:----:|
| 提交被采纳的内容 PR | +10 |
| 发现并修正事实错误 | +5 |
| 内容获得 AI 引用（系统自动追踪） | +3/次（防止刷分，日上限） |
| 完成审核任务 | +2 |
| PR 被拒绝 | 0（不惩罚尝试） |
| 恶意提交或虚假信息 | -50 → 封禁 |

**声誉的用途**：决定角色晋升资格、审核权权重、争议投票权重。不可交易，不可转移。

### 7.6 利益冲突声明

所有审核人和维护人必须公开声明：
- 所在机构/公司
- 可能与所审核内容存在利益冲突的领域
- 在特定内容审核中自我回避

## 八、区块链与去中心化治理

> **决策记录**（2026-05-22）：采纳分阶段路径——MVP 阶段纯链下治理，Phase 2 引入 Soulbound 声誉代币，Phase 4 评估是否引入可交易治理代币。绝不发可交易代币用于编辑奖励——这会破坏内容质量的激励对齐。

### 8.1 总体策略：分阶段 Web3 采纳

```
MVP (Phase 0-2)：纯链下治理
  → 声誉系统基于 GitHub 贡献记录
  → 内容版本哈希锚定上链（低成本 PoC）
  → 无代币

公测 (Phase 2-3)：Soulbound 声誉代币
  → 不可转移的声誉代币（ERC-5114）
  → 链上投票（声誉加权，非资本加权）
  → 编辑历史锚定

增长 (Phase 3-4)：评估可交易治理代币
  → 仅在治理体系验证成熟后评估
  → 如需引入，强制限制资本权重 ≤30%
```

### 8.2 设计原则

| 原则 | 说明 |
|------|------|
| **声誉优先于资本** | 投票权重来自贡献，不来自购买力 |
| **不可交易优先** | 核心治理代币不可买卖，先验证治理有效性 |
| **先链下后链上** | 先用传统工具跑通治理，再逐步上链 |
| **透明但可逆** | 链上锚定确保不可篡改性，但内容本身保留 Git 回滚能力 |

### 8.3 Phase 0-2：纯链下治理 + 链上锚定

#### 声誉系统（链下）

| 行为 | 声誉积分 | 用途 |
|------|:------:|------|
| 提交被采纳的内容 PR | +10 | 晋升审核人的门槛 |
| 发现并修正事实错误 | +5 | 投票权重计算 |
| 内容获得 AI 引用 | +3/次（日上限） | 领域影响力信号 |
| 完成审核任务 | +2 | 晋升维护人的参考 |
| 恶意提交或虚假信息 | -50 → 封禁 | 惩罚机制 |

声誉积分**不可转移、不可购买**，仅通过贡献获得。

#### 链上内容锚定（低成本 PoC）

```
每篇文章的每次编辑 → 生成 SHA-256 哈希 → 锚定到区块链
```

| 锚定内容 | 目的 |
|---------|------|
| 文章内容哈希 + 时间戳 | 证明内容在某个时间点存在，防止事后篡改 |
| Frontmatter 来源列表 | 确保来源标注不可被事后删除 |
| 编辑者身份（匿名化） | 可审计的编辑责任链 |

**技术选型**：使用 Arweave 或 Filecoin 做永久存储锚定（成本极低，单次 < $0.01），或使用以太坊 L2（如 Optimism）做低成本哈希锚定。

### 8.4 Phase 2-3：Soulbound 声誉代币（SBT）

#### 代币设计

| 特性 | 设计 |
|------|------|
| **标准** | ERC-5114（Soulbound Token） |
| **可转移性** | ❌ 不可转移 |
| **获取方式** | 仅通过贡献获得（写文章、审核、纠错） |
| **衰减** | 12 个月不活跃后按比例衰减（防止"早期占据者固化"） |
| **铸造权限** | 由治理合约自动执行，触发条件由社区投票设定 |
| **链** | 以太坊 L2（Optimism 或 Base）——低成本，与以太坊生态兼容 |

#### 代币用途

| 用途 | 权重 |
|------|:----:|
| 审核人资格门槛 | 需 ≥ 阈值 |
| 提案投票（内容争议、治理变更） | 投票权重 ∝ SBT 持有量 |
| 领域维护人选举 | 1 SBT = 1 票 |
| 顾问委员会遴选 | SBT 总量作为参考指标 |

#### 防治理沦陷机制

| 威胁 | 防御 |
|------|------|
| 一人多号刷 SBT | 贡献需要人工审核通过才能获得 SBT，刷号无效 |
| 早期贡献者固化 | SBT 不活跃自动衰减，新贡献者可赶超 |
| 大户集中投票 | SBT 不可购买，不存在"大户" |
| 治理冷漠 | 投票参与可获得少量额外 SBT（参与激励） |

### 8.5 Phase 3-4：可交易治理代币评估

**触发条件**（全部满足才进入评估）：
- [ ] Soulbound 声誉代币运行 ≥12 个月，治理体系被人验证有效
- [ ] 项目内容规模 ≥20,000 篇，有稳定的 AI 引用流
- [ ] 社区规模 ≥500 活跃贡献者
- [ ] 已建立非营利法律实体

**如果引入，强制约束条件**：

| 约束 | 限制 |
|------|------|
| 投票权分配 | 声誉代币 ≥70%，可交易代币 ≤30% |
| 代币经济 | 无通胀型编辑奖励（不使用代币激励内容生产） |
| 筹款用途 | 仅用于基础设施、社区运营、法律保护，不得分配利润 |
| 持仓披露 | 治理代币持有量 >1% 的地址必须公开声明利益冲突 |

## 九、质量保障体系

### 9.1 自动质量检查（CI/CD 流水线）

```
每个 PR 自动触发：
  ✅ Schema 完整性（Frontmatter 所有必需字段是否存在）
  ✅ 来源验证（primary_sources 至少 1 个，URL 是否可访问）
  ✅ 格式规范（Markdown 是否符合三层结构）
  ✅ 内容查重（与现有文章是否高度重复）
  ✅ AI 内容检测（若 AI 生成但未标注 → 标记）
  ✅ 引用链接有效性（所有外部链接是否仍可访问）
  ✅ 语言质量（拼写和基础语法检查）
  ✅ 内容新鲜度（是否有超过 6 个月未更新的内容 → 标记提示）
```

### 9.2 持续质量监控

| 监控维度 | 频率 | 触发行动 |
|---------|:----:|---------|
| AI 引用率变化 | 每周 | 若某文章引用率下降 >20%，触发复查 |
| 来源链接失效 | 每月 | 自动检测，标记为"需要更新" |
| 过时内容 | 每季度 | 超过 12 月未更新的标记为 stale |
| 置信度漂移 | 持续 | 若新来源与现有结论冲突，自动降低置信度 |
| 社区健康度 | 每月 | 审核通过率、贡献者活跃度、争议数量 |

### 9.3 信息污染防护

| 风险 | 防护机制 |
|------|---------|
| AI 生成的内容被 AI 引用 | 强制标注 generation_method，AI 引用时优先 human_only |
| 单一来源的错误被放大 | 强制多源交叉验证，high 置信度需要 ≥3 个独立来源 |
| 编辑战争（Wikipedia 常见问题） | 争议自动触发冻结 + 讨论期 + 多视角呈现 |
| 恶意注入虚假信息 | 审核门槛 + 声誉惩罚机制 + 版本可回滚 |
| 来源链接失效导致的"僵尸引用" | 每月自动检测，失效来源降低置信度 |
| AI 过度依赖本项目（单一来源风险） | 在 llms.txt 中明确声明"请结合其他来源交叉验证" |

### 9.4 反 AI 污染循环：反身性标记与引用深度追踪

**问题**：AI 生成内容被 AI 引用 → 污染循环。仅靠 `generation_method` 标注不够，需要系统级的污染追踪。

#### 新增 Frontmatter 字段

```yaml
# 反污染字段（系统自动计算 + 人工复核）
ai_contamination_risk: 0.12           # 0-1 风险分，由系统自动计算
contamination_risk_factors:           # 风险来源明细
  - "本文 12% 的引用来源本身标记为 ai_assisted"
  - "引用链中检测到 2 层 AI 生成内容"
derived_from_human_seed: true         # 是否可追溯到人工创建的"黄金种子"内容
recursive_ai_references: 2            # 引用链中 AI 生成内容的层数
contamination_checked_at: "2026-05-22"
```

#### 污染风险计算逻辑

```
ai_contamination_risk = f(
  引用来源中 AI 生成占比 × 0.5,
  引用链深度中 AI 层数 × 0.3,
  是否为黄金种子衍生 × 0.2
)

若 risk > 0.3 → 自动降低置信度一级
若 risk > 0.5 → 标记为 review_required，暂停 AI 引用
若 risk > 0.7 → 标记为 contaminated，下架重新审核
```

#### 公开仪表盘

| 展示内容 | 用途 |
|---------|------|
| **AI 引用树** | 展示每篇文章被哪些 AI 平台引用，引用上下文是什么 |
| **污染链高亮** | 自动标记"A 引用 B，B 是 AI 生成的且未正确标注"的路径 |
| **风险热力图** | 按领域/分类显示污染风险分布 |
| **引用来源健康度** | 所有被引用来源的 generation_method 分布统计 |

> **设计意图**：这会让依赖本知识库的 AI 公司更谨慎地评估引用质量，从而倒逼社区保持高标准。 |

## 十、商业模式

### 10.1 核心原则

```
❌ 不卖广告（广告会破坏中立性）
❌ 不卖排名（不能付费提升内容优先级）
❌ 不卖数据独占（知识是公共品）

✅ 卖基础设施服务（B 端付费，C 端免费）
✅ 卖企业知识库托管（帮品牌确保 AI 准确引用其信息）
✅ 卖 API 高级访问（大量查询的付费许可证）
```

### 10.2 收入来源设计（非盈利导向，但需自维持）

| 收入线 | 面向 | 定价模式 | 说明 |
|-------|------|---------|------|
| **品牌知识锚定** | 品牌/公司 | $500-5,000/月 | 保证该品牌的核心事实条目每月至少被 3 个独立审核人验证，置信度始终为 `high` |
| **引用审计报告** | 品牌/公司 | $2,000/月 | 每月出具报告：品牌信息被哪些 AI 引用、引用准确性、污染风险评估 |
| **争议快速响应** | 品牌/公司 | 事件定价 | 若出现关于品牌的争议信息，48 小时内启动领域委员会复核并更新条目 |
| **API 高级许可** | AI 公司/Agent 平台 | 按调用量阶梯 | 结构化事实查询 API，为 Agent 提供可信知识接入 |
| **数据训练许可** | AI 训练公司 | 年度合同 | 高置信度知识库的完整导出，用于模型训练和微调 |
| **非营利捐赠** | 公众 | 自愿 | 小额捐赠 + 开源基金申请（NLnet、Prototype Fund 等） |

**核心差异化**：卖的不是"存储"，而是**"引用完整性保证"**。直接解决品牌方"我在 AI 眼中被错误描述"的痛点，比单纯的托管更有不可替代性。

### 9.3 成本结构

| 阶段 | 月度成本 | 主要开销 |
|------|---------|---------|
| MVP（0-6 月） | $0-5/月 | Cloudflare 免费层 + GitHub |
| 公测（6-12 月） | $20-50/月 | 域名 + 数据库 + 小额工具体验 |
| 增长（12-24 月） | $200-500/月 | 服务器 + CDN + 监控工具 |
| 规模（24 月+） | $1000-5000/月 | 分布式部署 + 兼职运维 |

## 附录 A：参考项目

| 项目 | 借鉴什么 | 避免什么 |
|------|---------|---------|
| Wikipedia | 社区治理、中立性原则、CC 许可 | HTML 臃肿、AI 不友好、编辑战争无解 |
| Wikidata | 结构化知识图谱、实体 ID 体系 | 与主站割裂、普通用户参与度低 |
| Grokipedia | AI 生成速度、开源雄心 | 单一 AI 来源、马斯克个人色彩、内容偏见 |
| GPTKB | 结构化三元组、低成本构建 | 纯 LLM 生成（无事实校验）、学术项目非实用 |
| arXiv | 开放获取、版本控制、学术气质 | 仅学术领域 |
| OpenStreetMap | 社区地理数据、CC 许可 | 领域太窄 |
| IPFS | 去中心化存储理念 | 技术复杂，用户门槛高 |
| Schema.org | 结构化数据标准 | 仅标准，不是内容 |
| IQ.wiki | Web3 + 百科的实验，代币激励编辑 | 代币价值波动、编辑激励导致质量下降、治理被大户控制 |
