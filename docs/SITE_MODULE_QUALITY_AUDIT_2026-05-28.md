## Module Assessment

### Compiler and Static Site Generation

`src/compile.js` is now mostly an orchestration file. Frontmatter parsing,
record generation, HTML rendering, static output generation, content health,
coverage, examples, evals, graph, MCP metadata, and provenance artifacts live in
dedicated helpers under `src/lib/`.

The remaining compiler risk is contract drift, not size by itself. Future
compiler work should keep strengthening generated-output tests and avoid mixing
schema/path changes with internal refactors.

### Quality Gate and Article Eligibility

`src/quality-gate.js` and `src/lib/article-quality.js` are doing the right
kind of work: they separate fatal public eligibility from nonfatal quality
reasons. This lets draft content remain in the repository while preventing
explicitly published but ineligible articles from reaching the public surface.

The remaining gap is that some important nonfatal signals still require manual
audit review. Generic homepage sources, weak claim evidence, and low verified
coverage should continue to be repaired through sampled batches before any new
hard-fail policy is introduced.

### Public Audit Sampling

`scripts/audit-public-sample.js` remains useful as a human-readable review
artifact, while `npm run audit-public-full` is now the CI-grade public debt
gate. The full gate is the stop condition for public evidence debt: actionable
recommendations must stay at zero.

The next opportunity is presentation, not stricter gating. The content-health
artifact and production integrity report should be treated as the compact
machine summary, while the sampled audit remains the readable spot-check.

### AI Usefulness Benchmark

The representative query benchmark now has a local usefulness runner:
`npm run benchmark:ai`. It scores the built `dist/` artifacts through the same
context assembly path used by AI consumers, checking not only top-result routing
but also citation-ready claims, source URLs, source depth, and confidence.

The current benchmark passes at 100/100 across 17 representative queries and
reports no improvement candidates. It now covers citation-ready public topics,
natural-language AI intent routing, AnchorFact API usage guidance, and explicit
fallback behavior for live or time-sensitive requests that require external
sources.

### Verification Data

The current workflow deliberately separates local targeted repair from full
verification. `verification-report.json` should only be changed in bounded,
manual-review updates for the articles being repaired unless the scheduled
GitHub verification workflow is intentionally run.

The current source verification rate is still modest, and manual-reviewed
entries have grown through recent repair rounds. That is acceptable for the
current strategy, but the project should keep the weekly full verification
workflow as the long-term reconciliation mechanism.

### MCP Interfaces

The MCP stdio path and optional local HTTP wrapper now share the Python
manifest-backed article index and BM25 search implementation. The local
`anchorfact_context` tool also includes the same compact content-health summary
that production exposes through `/api/context`.

The remaining interface risk is cross-surface ranking semantics: the production
`/api/search` endpoint is backed by the richer signed static search index, while
the local MCP search is intentionally manifest-backed. That difference is
acceptable as long as `/api/evidence` and `/api/context` remain the recommended
one-call citation paths for AI consumers.

### Scripts and CI

The Node scripts are aligned with the project workflow:

- `npm test` checks confidence, article quality, compile behavior, integration,
  MCP index behavior, public audit logic, quality gate behavior, verification
  cache behavior, and repository hygiene.
- `npm run quality` protects public eligibility.
- `npm run build` produces the static output.
- `npm run audit-public-sample` refreshes the next public evidence sample.
- `npm run audit-public-full` enforces zero actionable public recommendations.
- `npm run production:integrity` verifies the live production trust chain.
- `npm audit` is clean.

CI runs tests, `pages:build`, and the full public audit gate on push and pull
request. A weekly/manual verification snapshot workflow runs `verify-full`,
quality, and build, then commits the refreshed verification report with
`[skip ci]`. This is a sensible split. A separate read-only Production Integrity
workflow now runs on schedule and can be dispatched manually to verify the live
signed trust chain after deployment.

### Documentation

`docs/CONTENT_STANDARD.md`, `docs/QUALITY_STANDARD.md`, README, launch
readiness, and the repair logs describe the practical content workflow well.
Repository hygiene now also checks documentation for stale launch metrics,
obsolete MCP tool-count narratives, and old benchmark case counts, so future
count or interface changes should update docs, smoke expectations, production
integrity expectations, and project-facing narratives together.

## Recommended Optimization Path

1. Make AI entrypoint choice even more obvious.
   Keep `/api/context`, `/api/evidence`, `/content-health.json`, `/coverage.json`,
   `/examples.json`, `/evals.json`, and `/mcp.json` aligned so agents can choose
   the right call without reading human documentation.

2. Turn the draft pool into a ranked repair queue.
   Use `/content-health.json` repair candidates to pick small batches from the
   387 draft articles, especially the lower-complexity candidates, without
   automatically publishing anything weak.

3. Improve benchmark candidates before broad expansion.
   Use `npm run benchmark:ai` after each build. If it reports low-confidence or
   single-source candidates, strengthen those public topics before publishing
   unrelated drafts. When it reports no candidates, return to the ranked draft
   repair queue rather than inventing new surfaces.

4. Keep monitoring trustworthy.
   Production Integrity should remain read-only, should not deploy or mutate
   Cloudflare state, and should fail only on real trust-chain, contract, count,
   or sustained availability problems.

5. Continue compiler and API refactors only behind behavior locks.
   Strengthen tests before reshaping internals, and do not change public output
   paths, schema versions, or signed artifact semantics in the same pass as a
   refactor.

6. Keep docs and launch metrics live.
   Count changes should update README, launch docs, content-health expectations,
   smoke expectations, production integrity expectations, and repo hygiene
   patterns together.
