- The public/draft split is stable after the recent repair rounds.

engines, LLM consumers, and direct article readers will see a technically
healthy site with uneven trust quality.

## Module Assessment

### Compiler and Static Site Generation

`src/compile.js` is effective but large. It currently handles content loading,
frontmatter parsing, confidence data, article JSON-LD, atomic facts, HTML, root
indexes, sitemap, robots, headers, dashboards, and filesystem writes in one
orchestration file.

This is acceptable while content quality is the highest-risk area, but it is the
largest maintainability target for a later engineering pass. The safest
refactor would first add behavior locks around generated output, then split
output writers and parsing helpers without changing public file shapes.

### Quality Gate and Article Eligibility

`src/quality-gate.js` and `src/lib/article-quality.js` are doing the right
kind of work: they separate fatal public eligibility from nonfatal quality
reasons. This lets draft content remain in the repository while preventing
explicitly published but ineligible articles from reaching the public surface.

The remaining gap is that some important nonfatal signals still require manual
audit review. Generic homepage sources, weak claim evidence, and low verified
coverage should continue to be repaired through sampled batches before any new
hard-fail policy is introduced.

### Public Audit Sampling

`scripts/audit-public-sample.js` is now the most useful operational tool for
quality improvement. Its fixed buckets catch low coverage, high-confidence
risk, medium capped claims, and low-confidence public articles without forcing a
full verifier run.

The process is working, but it is repetitive. After the next content batch,
consider extracting common evidence hygiene checks into shared helpers or
tests so the same source/claim issues are caught earlier.

### Verification Data

The current workflow deliberately separates local targeted repair from full
verification. `verification-report.json` should only be changed in bounded,
manual-review updates for the articles being repaired unless the scheduled
GitHub verification workflow is intentionally run.

The current source verification rate is still modest, and manual-reviewed
entries have grown through recent repair rounds. That is acceptable for the
current strategy, but the project should keep the weekly full verification
workflow as the long-term reconciliation mechanism.

### MCP Interfaces

The MCP stdio path and HTTP wrapper are useful, but they are not fully identical
interfaces. `src/mcp_index.py` provides the shared manifest-backed article index,
while `src/mcp_server.py` and `src/mcp_http.py` apply different search scoring
approaches. The stdio server uses a BM25-style search path, while the HTTP
wrapper has simpler keyword scoring.

This is not an immediate production blocker, but it is a real interface
consistency risk. A later module pass should share one ranking function or make
the intended difference explicit. `src/mcp_server.py` also contains
mojibake-looking tokenizer characters that should be cleaned under test.

### Scripts and CI

The Node scripts are aligned with the project workflow:

- `npm test` checks confidence, article quality, compile behavior, integration,
  MCP index behavior, public audit logic, quality gate behavior, verification
  cache behavior, and repository hygiene.
- `npm run quality` protects public eligibility.
- `npm run build` produces the static output.
- `npm run audit-public-sample` refreshes the next public evidence sample.
- `npm audit` is clean.

CI runs tests and `pages:build` on push and pull request. A weekly/manual
verification snapshot workflow runs `verify-full`, quality, and build, then
commits the refreshed verification report with `[skip ci]`. This is a sensible
split. The main missing release guard is that production smoke is currently a
manual post-push step rather than an automated deployment health gate.

### Documentation

`docs/CONTENT_STANDARD.md`, `docs/QUALITY_STANDARD.md`, and the repair logs now
describe the practical content workflow well. `docs/LAUNCH_READINESS_2026-05-27.md`
is partially stale because it records earlier public/draft/claim counts. It
should be refreshed after the next evidence repair batch or replaced with a
shorter rolling launch-status document.

## Recommended Optimization Path

1. Repair the current 12 sampled `repair_sources` articles.
   Keep scope limited to sampled public evidence, remove generic dispute text
   and weak homepage evidence, cap each article to source-mapped facts, update
   only targeted verification metadata, rebuild, audit, and keep `dist/`
   uncommitted.

2. Convert repeated evidence hygiene into reusable checks.
   After the current batch, identify repeated weak patterns from the audit logs
   and move the safest checks into shared hygiene helpers or targeted tests.
   Avoid turning every nonfatal signal into a hard quality-gate failure until
   enough sampled repairs prove the rule has low false positives.

3. Refactor compiler internals after behavior is locked.
   Split `src/compile.js` by responsibility only after tests cover the generated
   manifest, claims, sitemap, article pages, and draft behavior. Do not change
   public output paths or schemas in the same pass.

4. Align MCP search behavior.
   Share indexing and ranking behavior between stdio and HTTP interfaces, then
   add tests that prove both surfaces return consistent results for common
   queries, category filters, and missing-article cases.

5. Tighten release checks gradually.
   Keep the current CI shape, consider adding `npm audit` or a summarized public
   audit check to CI, and document when production smoke must run after deploys.
   Do not move `verify-full` into normal local repair work.
